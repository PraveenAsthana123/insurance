## What absence mode does

When operator is away, the §106 dispatcher (`scripts/auto_next_loop.py`)
runs every 5 minutes via cron. When uncommitted real-code is detected
AND absence mode is active, the dispatcher auto-commits files matching
the SAFE-pattern allowlist. This prevents the working tree from
accumulating un-reviewed parallel-session edits during operator absence.

## What absence mode does NOT do

- **NEVER pushes** — auto-commits stay local until operator returns or
  a separate scheduled push cron runs (see §Push policy below).
- **NEVER touches `backend/`, `scripts/`, `tests/drills/`, `.github/workflows/`,
  or any path containing `secrets`, `credentials`, `.env`, `.key`, `.pem`,
  `Dockerfile`, `docker-compose`**.
- **NEVER force-pushes** (§42 hard-gate · always operator-only).
- **NEVER auto-applies model-generated diffs** (per §50.7 — those still
  route through council review).
- **NEVER commits if §137 audit FAILS** (regression-gated).

## Safe pattern allowlist

The dispatcher auto-commits files matching ANY of these prefixes:

| Prefix | Rationale |
|---|---|
| `docs/*.md` | Documentation · low blast radius |
| `frontend/src/pages/bank/*.jsx` | Bank shell · §137 audit gates content |
| `frontend/src/pages/bank/tabs/*.jsx` | Bank tabs · §137 audit gates content |

## Unsafe token blocklist

ANY file path containing ANY of these tokens skips auto-commit (even with sentinel ON):

```
backend/ · scripts/ · tests/drills/ · .github/workflows/
secrets · credentials · .env · .key · .pem
Dockerfile · docker-compose
```

Default-deny: paths outside SAFE prefixes AND not matching UNSAFE tokens still skip (paranoid).

## Runtime churn filter

Always filtered (treated as auto-stageable runtime · not classification candidates):

```
data/prompt-history.md · data/work_tracker/ · data/registry/workforce_health.json
.agent/ · jobs/ · data/insurance/ · config/
```

## Per-commit gates (in order)

The handler runs ALL checks before committing:

1. Sentinel `.agent/absence-mode` exists
2. `git diff --name-only HEAD` succeeds
3. Filter runtime churn → real_changes list
4. Every real change is in SAFE_PREFIXES AND not in UNSAFE_TOKENS
5. `bash scripts/audit_no_black_backgrounds.sh` returns 0 (§137 PASS)
6. `git add` + `git commit -m <§51 substrate message>` succeed

If ANY check fails → handler returns `verdict: skip` with a reason · dispatcher records `status: skipped` · NO COMMIT made.

## Push policy during absence

The handler does NOT push. Two options for operator:

1. **Manual on return**: `git push insur-origin main` after operator is back · review accumulated commits before push.
2. **Scheduled cron**: (NOT installed by default · operator decision)
   ```cron
   # Push absence-mode commits every 4 hours · only when sentinel present
   0 */4 * * * cd /mnt/deepa/insur_project && \
     [ -f .agent/absence-mode ] && \
     git log --oneline origin/main..HEAD | grep -q "absence-mode auto-integrate" && \
     git push insur-origin main \
     >> jobs/logs/absence_mode_push.log 2>&1
   ```
   Operator opts in by adding the line to their crontab.

## Monitoring during absence

Check accumulated state when returning:

```bash
# How many absence-mode commits accumulated?
git log --oneline --grep="absence-mode auto-integrate" insur-origin/main..HEAD | wc -l

# What files were touched?
git log --grep="absence-mode auto-integrate" insur-origin/main..HEAD --stat

# Recent dispatcher activity (status histogram)
ls -t jobs/reports/auto-next/run-*.json | head -50 \
  | xargs grep -h '"status"' \
  | sed -E 's/.*"status"[^"]*"([^"]+)".*/\1/' \
  | sort | uniq -c

# Any skipped attempts (reasons for refusal)?
ls -t jobs/reports/auto-next/run-*.json | head -50 \
  | xargs grep -l '"skipped"' \
  | xargs grep -h '"reason"' | head
```

## Brutal rule

> Absence mode is **opt-in** (sentinel file) and **conservative** (default-deny
> classification · multiple gates · auto-commit ONLY for documentation +
> bank-shell content with §137 audit PASS). Anything outside this narrow
> path falls through to operator review on return. The autonomous loop
> gains ZERO new authority during absence — it just exercises the SAFE
> subset of the existing safe-allowlist without operator typing `next`.
